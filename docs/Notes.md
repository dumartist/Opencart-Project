## 1. SSH Key Authentication Issues

### âŒ Problem
After deploying infrastructure, couldn't SSH from bastion to web servers:
```bash
ubuntu@bastion:~$ ssh ubuntu@10.0.101.18
Permission denied (publickey).
```

### ğŸ” Root Cause
SSH key authentication requires:
- **Public key** â†’ on the server (web servers had this via `aws_key_pair`)
- **Private key** â†’ on the client (bastion did NOT have this!)

The bastion only had the public key registered. When SSHing to web servers, it had no private key to authenticate with.

### âœ… Solution
Auto-generate SSH key pair in Terraform and inject private key into bastion:

```hcl

  algorithm = "RSA"
  rsa_bits  = 4096
}

resource "aws_key_pair" "bastion" {
  key_name   = "bastion-key"
  public_key = tls_private_key.bastion.public_key_openssh
}

# Bastion user_data installs private key
resource "aws_instance" "bastion" {
  user_data = <<-EOF
    # ... other setup ...
    
    # Install private key for SSH access to web servers
    mkdir -p /home/ubuntu/.ssh
    cat > /home/ubuntu/.ssh/id_rsa << 'KEYEOF'
${tls_private_key.bastion.private_key_pem}
KEYEOF
    chmod 600 /home/ubuntu/.ssh/id_rsa
    chown -R ubuntu:ubuntu /home/ubuntu/.ssh
  EOF
}
```

## 1b. SSH Key Mismatch - Local bastion-key File Issue

### âŒ Problem
After redeploying infrastructure, SSH to bastion fails:
```powershell
PS> ssh -i bastion-key ubuntu@3.231.168.48
ubuntu@3.231.168.48: Permission denied (publickey).
```

### ğŸ” Root Cause
The local `bastion-key` file was from a **previous deployment** and didn't match the new key generated by Terraform. Even though the file contents *looked* similar (both started with `MIIJKAIBAAKCAgEA5JGeKs9I...`), they were completely different keys!

**How to verify (fingerprints must match):**
```powershell
# Terraform's key fingerprint
terraform output bastion_key_fingerprint
# Output: SHA256:jKhNKHnXIDmz7lISkO3n3+XYPPzl/vB37woei5ywJvI

# Local file fingerprint
ssh-keygen -lf bastion-key
# Output: SHA256:DwilOJIaXNVHMHc1VJ9dQ/3oD7wj9xZ4Pldeag9jmmE  â† DIFFERENT!
```

### âœ… Solution
Added a `null_resource` with `local-exec` to **automatically export** the SSH key after `terraform apply`:

```hcl
resource "null_resource" "save_bastion_key" {
  depends_on = [tls_private_key.bastion]

  triggers = {
    key_id = tls_private_key.bastion.id  # Re-run if key changes
  }

  provisioner "local-exec" {
    command     = <<-EOT
      # Save private key
      $privateKey = @'
${tls_private_key.bastion.private_key_pem}
'@
      $privateKey | Set-Content -Path "bastion-key" -NoNewline -Encoding ASCII
      icacls bastion-key /inheritance:r /grant "$($env:USERNAME):R" | Out-Null
      
      # Save public key (CRITICAL: prevents OpenSSH from using stale .pub file)
      $publicKey = "${tls_private_key.bastion.public_key_openssh}"
      $publicKey | Set-Content -Path "bastion-key.pub" -NoNewline -Encoding ASCII
      
      Write-Host "[SUCCESS] Bastion SSH keys saved"
    EOT
    interpreter = ["PowerShell", "-Command"]
  }
}
```

### âš ï¸ Why `terraform output -raw` Doesn't Work on Windows
PowerShell's output redirection (`>`) causes encoding issues:
- Adds UTF-16 BOM
- Converts `\n` to spaces
- Creates invalid key format: `Load key "bastion-key": invalid format`

**Manual fix if auto-export fails:**
```powershell
$s = Get-Content terraform.tfstate -Raw | ConvertFrom-Json
$s.outputs.bastion_private_key.value | Set-Content bastion-key -NoNewline
icacls bastion-key /inheritance:r /grant "$($env:USERNAME):R"
# Also regenerate .pub file from private key:
ssh-keygen -y -f bastion-key > bastion-key.pub
```

## 1c. SSH Key - Stale .pub File Issue (OpenSSH Caching)

### âŒ Problem
After redeploying, the `bastion-key` private key was updated correctly, but SSH still failed:
```powershell
PS> ssh -i bastion-key ubuntu@3.231.168.48
ubuntu@3.231.168.48: Permission denied (publickey).
```

**Confusing behavior:** A freshly copied key with a different name worked fine:
```powershell
PS> ssh -i bastion-key-fresh ubuntu@3.231.168.48
bastion-key-fresh works!  # â† SUCCESS!
```

Both files were **byte-for-byte identical** (same SHA256 hash), yet one worked and one didn't!

### ğŸ” Root Cause
**OpenSSH on Windows automatically reads `.pub` files alongside private keys!**

When you use `-i bastion-key`, OpenSSH looks for `bastion-key.pub` next to it and uses that for authentication.

| File | Content | Problem |
|------|---------|---------|
| `bastion-key` | âœ… Correct (Terraform-generated) | - |
| `bastion-key.pub` | âŒ **OLD** (from previous deployment) | Stale! |
| `bastion-key-fresh` | âœ… Correct (same content) | - |
| `bastion-key-fresh.pub` | ğŸš« Does not exist | SSH derives from private key |

**How `ssh-keygen -l` can lie:**
```powershell
# With stale .pub file present:
ssh-keygen -l -f bastion-key
# Output: SHA256:DwilOJIaXNVHMHc1VJ9dQ...  â† Reading from .pub file!

# After removing/renaming .pub file:
ssh-keygen -l -f bastion-key
# Output: SHA256:jKhNKHnXIDmz7lISkO3n3...  â† Correct (derived from private key)
```

### âœ… Solution
**Updated `null_resource` to save BOTH private AND public keys:**

The Terraform script now exports both files together, ensuring they always match:
- `bastion-key` - Private key (with restricted permissions)
- `bastion-key.pub` - Public key (regenerated from Terraform's key)

**Manual fix if you encounter this:**
```powershell
# Option 1: Regenerate .pub from private key
ssh-keygen -y -f bastion-key > bastion-key.pub

# Option 2: Delete the .pub file entirely (SSH will derive it)
Remove-Item bastion-key.pub

# Option 3: Use a different key filename
Copy-Item bastion-key my-key
ssh -i my-key ubuntu@<ip>
```

### ğŸ’¡ Lesson Learned
On Windows, **always keep private key and .pub file in sync**. If you update one, update the other. The auto-export script now handles this automatically.

## 2. AWS Backup Lifecycle Restrictions

### âŒ Problem
Terraform apply failed with error:
```
Error: creating Backup Plan: InvalidParameterValueException: 
DeleteAfterDays cannot be less than 90 days apart from MoveToColdStorageAfterDays
```

### ğŸ” Root Cause
AWS Backup has a **minimum 90-day gap requirement** between moving to cold storage and deletion. Our original config:
```hcl
lifecycle {
  cold_storage_after = 7    # Move to cold after 7 days
  delete_after       = 30   # Delete after 30 days
  # Gap = 23 days âŒ (must be â‰¥90)
}
```

### âœ… Solution
For short-term backups (daily), removed cold storage entirely:

```hcl
# Daily backup - no cold storage (short retention)
rule {
  rule_name = "daily-efs-backup"
  lifecycle {
    delete_after = 30  # Just delete, no cold storage
  }
}

# Weekly backup - longer retention, still no cold storage
rule {
  rule_name = "weekly-efs-backup"
  lifecycle {
    delete_after = 90
  }
}
```

### ğŸ’¡ Lesson Learned
Cold storage only makes sense for **long-term archival** (â‰¥90 days retention). For disaster recovery with shorter retention, just use standard storage.

## 4. ElastiCache/Redis - Gave Up

### âŒ Problem
Initially considered Redis for:
- Session management across ASG instances
- Database query caching
- Shopping cart persistence

### ğŸ” Challenges
1. **Cost**: ElastiCache adds monthly charges
2. **Complexity**: Additional infrastructure to manage
3. **Learner Lab**: Limited budget/credits

### âœ… Alternative Solution
Used **Cloudflare CDN + ALB Sticky Sessions** instead:

```hcl

    type            = "lb_cookie"
    cookie_duration = 86400  # 24 hours
    enabled         = true
  }
}
```

**How it works:**
- User's first request â†’ routed to any web server
- ALB sets a cookie â†’ "sticks" user to that server
- All subsequent requests â†’ same server (24 hours)
- PHP sessions stored locally (file-based)

### ğŸ“Š Comparison

| Aspect | Redis | Sticky Sessions |
|--------|-------|-----------------|
| Cost | ğŸ’° Higher | âœ… Free |
| Complexity | Complex | Simple |
| Session Survival | âœ… Survives instance death | âŒ Lost if instance dies |
| Best For | High traffic, critical sessions | Learning, small-medium sites |

### ğŸ’¡ Verdict
**Gave up on Redis** - Sticky sessions sufficient for learning environment and small e-commerce.

## 9. Sensitive Variables Security

### âŒ Original Problem
Hardcoded credentials in `main.tf`:
```hcl
# BAD - exposed in version control!

  db_password = "MySecretPassword123!"
}
```

### âœ… Solution: Variable Separation

**Created three files:**

1. **variables.tf** - Declarations (safe to commit)
```hcl

  description = "Database password"
  type        = string
  sensitive   = true
}
```

2. **terraform.tfvars** - Actual values (in .gitignore!)
```hcl
db_password = "ActualSecretPassword123!"
```

3. **terraform.tfvars.example** - Template for others
```hcl
db_password = "CHANGE_ME_strong_password_here"
```

### ğŸ“ .gitignore
